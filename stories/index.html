User Stories — Spring Music / Northwind Logistics Catalog

PM/BA Deliverable • Scenario 1: Code Modernization • The Stories Waypoint

Northwind Logistics runs a catalog service built on Spring Boot 2.4. The board approved "modernization" without defining it. These stories capture what the system must continue to do correctly through any modernization effort, and where stakeholders disagree about scope. Stories are written against the current API surface so they can be used directly as a basis for characterization tests (The Pin) before any refactoring begins.

Personas

Catalog UserInternal staff browsing the album/catalog data
Catalog CuratorStaff who add, edit, or remove catalog entries
Ops EngineerResponsible for deployment, monitoring, and uptime
DeveloperEngineer working on the codebase
CAP-1

Browse the Album Catalog

CAP-1-S1
View the full catalog
▼
As a Catalog User I want to retrieve a list of all albums in the system so that I can see everything currently in the catalog
Acceptance Criteria
  1. 1GET /albums returns HTTP 200
  2. 2The response body is a JSON array
  3. 3Each element contains at minimum: id, title, artist, releaseYear, genre, trackCount
  4. 4When the catalog has been pre-populated on first startup, the response contains at least 20 albums
  5. 5An empty catalog returns an empty JSON array [], not a 404 or error
Out of scope: Pagination, filtering, sorting
CAP-1-S2
Retrieve a single album by ID
▼
As a Catalog User I want to fetch one album by its unique identifier so that I can view its full details without loading the entire catalog
Acceptance Criteria
  1. 1GET /albums/{id} with a valid existing ID returns HTTP 200 and the album object
  2. 2The response contains all fields: id, title, artist, releaseYear, genre, trackCount, albumId
  3. 3GET /albums/{id} with a non-existent ID returns HTTP 404
  4. 4GET /albums/{id} with a malformed or empty ID returns HTTP 400 or 404 (not 500)
Out of scope: Fuzzy lookup by title or artist
CAP-2

Add / Update / Remove Albums

CAP-2-S1
Add a new album
▼
As a Catalog Curator I want to add a new album to the catalog so that the collection stays current
Acceptance Criteria
  1. 1PUT /albums with a valid JSON body returns HTTP 200 or 201
  2. 2The response body includes the newly created album with a system-generated id (UUID format)
  3. 3The new album is retrievable via GET /albums/{id} immediately after creation
  4. 4PUT /albums with a missing required field returns HTTP 400, not 500
  5. 5Two successive PUT /albums calls with identical data create two distinct albums with different IDs
Out of scope: Duplicate detection, image upload
Stakeholder Disagreement — Priority

Dev team wants this story deprioritized until the multi-DB profile layer (CAP-3) is stabilized first.

Business wants CAP-2 as the first extraction target because curators are blocked by the current UI.

Status: UNRESOLVED — The decomposition plan (The Map) must address this before sprint planning.

CAP-2-S2
Update an existing album
▼
As a Catalog Curator I want to correct or update metadata on an existing album so that the catalog reflects accurate information
Acceptance Criteria
  1. 1POST /albums with a valid JSON body containing an existing id returns HTTP 200
  2. 2Updated fields are reflected in subsequent GET /albums/{id} responses
  3. 3Fields not included in the request body are not cleared or nulled out
  4. 4POST /albums with a non-existent id returns HTTP 404, not a silent insert
  5. 5Updating id itself is rejected (HTTP 400) — IDs are immutable
Out of scope: Partial patch (PATCH), field-level audit trail
Stakeholder Disagreement — Test Coverage Gate

QA requires characterization tests (The Pin) to be green before any changes to the update path.

PM has delivery pressure to ship the curator UI fix this sprint.

Status: UNRESOLVED — Recommendation: QA's gate wins. A broken update path in prod is worse than a delayed sprint.

CAP-2-S3
Delete an album
▼
As a Catalog Curator I want to remove an album from the catalog so that the collection only contains relevant entries
Acceptance Criteria
  1. 1DELETE /albums/{id} with a valid existing ID returns HTTP 200 or 204
  2. 2Subsequent GET /albums/{id} for the deleted album returns HTTP 404
  3. 3The deleted album no longer appears in GET /albums
  4. 4DELETE /albums/{id} with a non-existent ID returns HTTP 404, not 200
  5. 5A second DELETE on the same ID returns 404 (not 200 or 500)
Out of scope: Soft delete, archive, undo
CAP-3

Database Backend Flexibility

CAP-3-S1
App starts and serves data against any supported backend
▼
As an Ops Engineer I want the application to function correctly regardless of which database backend is bound so that I can deploy to any environment without code changes
Acceptance Criteria
  1. 1With no active profile (default/H2), app starts and GET /albums returns data
  2. 2With profile mysql, app connects to MySQL and GET /albums returns data
  3. 3With profile mongodb, app connects to MongoDB and GET /albums returns data
  4. 4With profile redis, app connects to Redis and GET /albums returns data
  5. 5Activating more than one database profile simultaneously causes a clear startup failure (not silent data inconsistency)
  6. 6The API response structure of GET /albums is identical regardless of which backend is active
Out of scope: Live migration of data between backends, simultaneous multi-backend writes
Stakeholder Disagreement — "Modernize" Means What?

CTO interprets modernization as decomposing the multi-profile capability into separate microservices behind an API gateway.

Ops says: containerize the monolith as-is and ship it — profile switching is a feature, not a problem.

Status: UNRESOLVED — Highest-impact disagreement. The ADR (The Map) must make a binding decision before CAP-3 is touched.

CAP-4

Sample Data Initialization

CAP-4-S1
Catalog is pre-populated on first startup
▼
As a Catalog User I want the catalog to contain sample data when first deployed so that the system is immediately usable for demonstration and onboarding
Acceptance Criteria
  1. 1On first startup against an empty database, GET /albums returns exactly 28 albums
  2. 2Sample albums include: The Beatles, The Rolling Stones, Led Zeppelin, Nirvana (spot-check)
  3. 3On subsequent restarts against a non-empty database, no duplicate albums are added
  4. 4If a curator has deleted albums, a restart does not restore the deleted items
  5. 5Initialization completes before the first HTTP request is served — no race condition window where GET /albums briefly returns []
Out of scope: Seeding with production data, environment-specific seed sets
Stakeholder Disagreement — Sample Data in Production

Business wants sample data removed from production deployments — classic rock albums in a logistics catalog looks unprofessional.

Dev wants to keep it — it's the only reliable way to demo the system end-to-end.

Recommended resolution: Make seed data opt-in via a Spring profile (--spring.profiles.active=seed). Neither side loses. Add as a story next sprint.

CAP-5

Application Health Visibility

CAP-5-S1
Ops can see which database profile is active at runtime
▼
As an Ops Engineer I want to query the running application and know which database backend and Spring profiles are active so that I can confirm the correct configuration was applied after deployment
Acceptance Criteria
  1. 1GET /appinfo returns HTTP 200
  2. 2The response includes a profiles field listing all active Spring profiles as an array
  3. 3When running with the H2 default, profiles reflects that (empty array or ["default"] is acceptable)
  4. 4When running with mysql profile, profiles contains "mysql"
  5. 5The response includes a services field — may be empty in local deployments but must be present
  6. 6GET /appinfo is available without authentication
Out of scope: Prometheus metrics, distributed tracing, log aggregation
